# Spec: gentle-engram-cloud-autotick

## Change metadata

- Change: `gentle-engram-cloud-autotick`
- Capability: `pi-cloud-autosync-parity`
- Kind: MODIFIED Pi package behavior
- REQ range: REQ-PI-AUTOSYNC-001 through REQ-PI-AUTOSYNC-006

## Purpose

Pi `gentle-engram` MUST make Engram Cloud autosync work automatically for Pi-managed Engram processes when Cloud is configured, without moving sync behavior into the plugin.

## Requirements

### REQ-PI-AUTOSYNC-001: Auto-enable child autosync from env Cloud config

When the Pi package launches an Engram child process and `ENGRAM_CLOUD_AUTOSYNC` is unset/blank, and both `ENGRAM_CLOUD_TOKEN` and `ENGRAM_CLOUD_SERVER` are non-empty, it MUST pass `ENGRAM_CLOUD_AUTOSYNC=1` to the child process.

**Scenarios**:

- GIVEN `ENGRAM_CLOUD_TOKEN=tok` and `ENGRAM_CLOUD_SERVER=https://cloud.example`, WHEN the Pi helper builds the child env, THEN `ENGRAM_CLOUD_AUTOSYNC` is `1`.
- GIVEN token is missing, WHEN the helper builds the child env, THEN `ENGRAM_CLOUD_AUTOSYNC` is absent/unchanged.

### REQ-PI-AUTOSYNC-002: Auto-enable from persisted cloud.json server

When `ENGRAM_CLOUD_SERVER` is unset but `${ENGRAM_DATA_DIR:-~/.engram}/cloud.json` contains non-empty `server_url`, and `ENGRAM_CLOUD_TOKEN` is non-empty, the Pi package MUST pass `ENGRAM_CLOUD_AUTOSYNC=1` to child Engram processes.

**Scenarios**:

- GIVEN `ENGRAM_DATA_DIR=/tmp/data`, `/tmp/data/cloud.json` contains `{"server_url":"https://cloud.example"}`, and token is set, WHEN the helper builds env, THEN autosync is enabled.
- GIVEN `cloud.json` is absent or malformed, WHEN the helper builds env, THEN autosync is not auto-enabled unless `ENGRAM_CLOUD_SERVER` is set.

### REQ-PI-AUTOSYNC-003: Explicit autosync env is authoritative

The Pi package MUST NOT override a non-blank `ENGRAM_CLOUD_AUTOSYNC` value.

**Scenarios**:

- GIVEN `ENGRAM_CLOUD_AUTOSYNC=0`, token, and server, WHEN the helper builds env, THEN the value remains `0`.
- GIVEN `ENGRAM_CLOUD_AUTOSYNC=1`, WHEN the helper builds env, THEN the value remains `1`.

### REQ-PI-AUTOSYNC-004: Pi HTTP server auto-start uses autosync env helper

When `plugin/pi/index.ts` auto-starts `engram serve`, it MUST use the helper-generated child environment.

**Scenario**:

- GIVEN source inspection of `plugin/pi/index.ts`, WHEN `spawn()` starts Engram, THEN the spawn options include helper-generated env rather than bare inherited environment.

### REQ-PI-AUTOSYNC-005: Pi MCP launcher uses equivalent autosync gate

The MCP launcher generated by `pi-engram init` and the packaged `mcp-template.json` MUST include equivalent token+server gated autosync env logic before starting `engram mcp --tools=agent`.

**Scenario**:

- GIVEN source inspection of `plugin/pi/cli.js` and `plugin/pi/mcp-template.json`, WHEN reviewing the launcher, THEN it contains `ENGRAM_CLOUD_AUTOSYNC` auto-enable logic and still launches `engram mcp --tools=agent`.

### REQ-PI-AUTOSYNC-006: Documentation explains ownership and opt-in

Docs MUST state that Pi `gentle-engram` auto-enables the existing Engram core autosync manager for Pi-launched/configured Engram processes when Cloud token+server are configured. Docs MUST also state that the 30-second tick remains in Engram core and that project enrollment/server policy still controls what syncs.

**Scenario**:

- GIVEN `plugin/pi/README.md` and `docs/AGENT-SETUP.md`, WHEN a user reads Cloud autosync setup, THEN they can distinguish Pi auto-enable from raw CLI env setup.
